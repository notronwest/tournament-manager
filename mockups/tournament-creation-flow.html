Mockup — Tournament Creation

Wizard for going from blank slate to a publishable (or draft-able) tournament. Friendly to the first-time organizer (skip-able steps, inline hints), efficient for the returning organizer (clone + default event template), and never trapping the user — Save as draft is always one click away.

Briefs being drawn from: docs/scenarios/tournament-lifecycle.md — scenarios C1 (Ron / clone), C2 (Sarah / first-timer), C3 (Dave / multi-event), C5 (Cancellation policy preset). Backlog items above ("Creation wizard," "Clone a tournament," "Cancellation Policy as a creation-wizard step") get their UI from this mockup.
Defaults baked into this mockup. Push back on any of these.
  • Wizard, not single form. Left rail with 8 steps; right pane is the active step. Per-step URLs (e.g. /admin/:org/tournaments/new/events) so deep-link / refresh / bookmark all work.
  • Wizard replaces create-mode of TournamentFormPage entirely. Edit-mode stays as the existing all-fields form for now.
  • Save & exit at any step routes back to the Tournaments list. Re-entering opens the wizard at the next unfinished step.
  • Autosave per step. The user never loses progress; the save indicator in the rail confirms that.
  • "What does this mean?" collapsible hints next to fields that use jargon, so a first-timer doesn't need to leave the wizard to look something up.
  • Default event template on Step 2 — typical club bracket pre-populated. The organizer adjusts rather than enters from scratch. Per-event Edit is inline-expand on the row (matches the public-flow pattern).
  • Clone deep-copies everything and lands in the Review step with all fields pre-filled (just verify the new dates).
  • Publish is reachable from the rail but stays disabled until the hard-required set is met — see "Minimum to publish" callout below.
Minimum to publish (decided):
  • Hard-required (Publish stays disabled until met): tournament name, start + end date, location, ≥1 event, and pricing decision made (any value, including $0 for a free tournament).
  • Soft-required (Publish allowed, but a confirm modal calls out what's missing): cancellation policy (public page falls back to "contact organizer" if unset) and Stripe Connect (registrations save as paid without collecting money — fine for testing, not for production).
  • Optional: sponsors / branding, FAQs.
Hard-required set drives the Publish-button enabled state on every screen of the wizard. Review screen surfaces both hard and soft gaps as concrete checklist items.

Why are these the defaults?
Most club tournaments run the same basic bracket year over year: men's / women's / mixed doubles across the common skill tiers (3.0, 3.5, 4.0+). If you want a leaner tournament, just delete the rows you don't want. If you want different skill breakpoints (2.5, 4.5, 5.0), edit any row's rating range.
Default-template + checkbox adjust beats blank slate. The table is pre-filled with the typical doubles divisions plus the custom "Open Singles" Dave added. Checkboxes drive inclusion — checked = will be created when the tournament publishes; unchecked = excluded and faded out (still visible so the user can re-check easily). Per-row Edit is an inline-expand on the row (same pattern as the public-flow register card) that reveals format / gender / rating range / capacity / fees / bracket type controls without leaving the page. Skip is available — a tournament with no events saves as draft; Publish stays disabled until at least one event is checked.
Open question: the default template is "typical club bracket." Should it be configurable per organization (e.g. WMPC always uses 2.5/3.0/3.5/4.0/4.5, another club starts at 3.0)? An org-level default-events setting would mean a new club's first tournament walks through this once and never has to again. Probably a follow-up after the basic version ships.
Step 3 has four variants — pick the pattern that matches how you run pricing. Most tournaments fall into one of four buckets:

Single price — one price for everyone, no dates. The default and the simplest case.
Early bird — discount up to a date, regular after. Classic incentive to commit early.
Early bird + Late fee — three windows with a surcharge close to the start. Some clubs use the late fee to discourage day-of registrations that throw off bracket sizes.
Custom — N tiers with your own labels. Escape hatch for unusual cases (members-only pre-sale, two-stage discounts, etc.). Hopefully rarely used.

The pattern picker at the top of Step 3 hides the date inputs entirely when "Single price" is chosen — the simple case stays simple. The next four screens show what each variant looks like in the wizard.

One concept, two surfaces. These same tier dates drive what the public tournament page advertises ("Early bird ends in 4 days," "Late registration starts Jul 6") and the public lifecycle status pill (Early Bird Registration Open → Registration Open → Late Registration Open) — organizers don't manage status flags separately from pricing.

Price locks at checkout. When a player commits, the active tier's price is snapshotted onto their registration — so someone who registered during the Early Bird window keeps that price even if the deadline passes before they pay or before the tournament starts. No "wait, why did the price change on me?" surprises.

Adding a third tier costs almost nothing. The same UI shape scales from one tier to three — same input rows, same preview-math bands, just more of them. And if even three isn't enough, the Custom variant on the next screen takes it to N.

Open question: what's the smallest blocking set before Publish actually does anything? Definitely: name + dates + ≥1 event. Probably: location, otherwise the public page reads weirdly. Open questions — resolved.
  • Routing: per-step URLs. /admin/:org/tournaments/new/basics → /events → /pricing → … Refresh-safe, bookmark-able, "come back tomorrow" works.
  • Save & exit: always routes to the Tournaments list; re-entering the draft tournament opens the wizard at the next-unfinished step.
