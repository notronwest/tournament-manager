Issue #57 — Centralize the "Register" button

Three design approaches for the doubles inline-register flow in PublicTournamentPage. Each section shows the same states: no partner picked · partner picked · already registered (for reference).

Option A Contextual label — smallest change

Keep the current per-event → checkout flow but rename the button to match its actual consequence. No partner picked: button is disabled with a hint. Partner picked: button clearly says where you're going and what's being saved. Minimal code change — one conditional on the button label.

Variants: "Continue to checkout →" (neutral, no mention of partner)  ·  "Save & pay" (shorter)  ·  "Confirm & checkout →" (emphasizes confirmation)

Option B Cart model — "Save" per event, one checkout at the end

The per-event button becomes "Save" (or "Add to registration") — it commits the event + partner but does not immediately navigate to checkout. Instead, a summary bar/section appears below all events once you have at least one saved item, and a single "Complete Registration" CTA handles checkout for everything at once. This is the model RegisterPage already uses (Confirm X changes → checkout).

Option C Sticky bottom bar — always-visible checkout CTA

Like Option B but the checkout CTA lives in a sticky footer bar instead of inline below the events. The bar appears as soon as one event is saved, stays visible while the player continues picking events, and shows a running total. Familiar from e-commerce (cart drawer / floating bag). The per-event button still says "Save"; the sticky bar says "Go to checkout" with a badge showing item count.

Tradeoff vs Option B: The sticky bar is always visible regardless of how far down the page the player is, which helps when there are many events. But it's a bigger implementation — requires a floating overlay positioned relative to the card container. For a short list of events it may feel like overkill.

Summary Trade-off at a glance
Per-event CTA label Checkout trigger Multi-event checkout? Implementation
A "Save partner & checkout →" Immediately after clicking per-event CTA No — one event at a time Tiny — rename string only
B "Save" "Complete Registration" bar below events Yes — all saved events at once Medium — cart bar component + state
C "Save" "Go to checkout" sticky footer bar Yes — all saved events at once Larger — sticky overlay + position logic

Key open question for B & C: When "Save" is clicked on a pending-payment flow, does it create the event_registration row immediately (with status='pending_payment') or does it live purely in client state until "Complete Registration" is clicked? Creating rows immediately is safer (partner invite goes out, capacity slot is held) but means partial saves are possible if the player never checks out.
